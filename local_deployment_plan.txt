这是你这个系统走向“可交付产品”的关键问题之一：**本地部署时，大模型怎么放进去**。

先给结论：

> 工业场景里，大模型不是“中心”，而是一个**可替换的推理服务层（Inference Layer）**，并且必须支持“离线 + 可降级 + 可混合”。

---

# 一、先拆清楚：你在系统里到底需要大模型做什么

在你的DWG识别系统中，大模型主要做三类事：

## 1️⃣ 语义理解（必须）

* DCC / Rack / PLC 是什么
* block 名字映射到设备类型
* text → tag / attribute

---

## 2️⃣ 关系推理（非常重要）

* DCC 是否 mounted_on Rack
* PLC 是否属于 cabinet
* 设备层级归属

---

## 3️⃣ 图纸“脏数据补全”

* 图层乱
* block 不规范
* 标签缺失
* OCR文本歧义

---

# 二、核心现实：本地部署不能依赖云大模型

工业客户有几个硬约束：

* ❌ 不能联网（内网）
* ❌ 数据不能出厂
* ❌ 延迟要稳定
* ❌ 成本不能按token计费
* ❌ 必须可控、可审计

所以：

> OpenAI / Claude API 在工业交付里 = 只能用于研发，不能用于交付核心路径

---

# 三、工业级标准架构（重点）

你应该把系统拆成 3 层：

```text id="v8qk1m"

                │  应用层（UI/API） │
                └────────┬─────────┘
                         │
        ┌────────────────┼────────────────┐
        ▼                ▼                ▼
  规则引擎         空间引擎         LLM推理层
 (deterministic)  (PostGIS)      (AI reasoning)
```

# 五、方案2：LLM + 规则混合（工业推荐）

这是最关键的设计思想：

> ❗ 不要让LLM做“决定”，只让它做“建议”

---

## 架构：

```text id="h7qk2a"
          DWG解析
             ↓
      结构化数据（Object）
             ↓
   ┌─────────┴─────────┐
   ▼                   ▼
规则引擎           LLM推理
（强约束）        （弱推理）
   └─────────┬─────────┘
             ▼
        最终关系图
```

---

## 举例：

### 判断：

```text id="n3xq7d"
DCC 是否 mounted_on Rack
```

---

### 规则优先：

```text id="j8wq2m"
如果距离 < 50mm 且上下对齐
→ 直接成立
```

---

### LLM只做：

```text id="p2xq8s"
如果规则不确定：
→ 给概率建议
```

# 七、关键设计：LLM不能直接访问DWG

这是很多人会犯的错误。

❌ 错误方式：

```text id="k2wq9s"
LLM 直接读DWG
```

---

✅ 正确方式：

```text id="x8qk3m"
DWG → Object Store → LLM只看结构化JSON
```

---

### LLM输入应该长这样：

```json id="n7wq2p"
{
  "objects":[
    {
      "id":"DCC01",
      "bbox":[100,200,300,400],
      "layer":"EQUIPMENT"
    },
    {
      "id":"RACK01",
      "bbox":[100,410,300,600],
      "layer":"SUPPORT"
    }
  ]
}
```

---

### LLM输出：

```json id="m3wq8d"
[
  {
    "from":"DCC01",
    "to":"RACK01",
    "relation":"mounted_on",
    "confidence":0.86
  }
]
```
